# Investigation: Synchronized Multi-M4A Audio and Transcript Tool Architecture

## Executive Summary

This investigation compares four viable TypeScript application architectures for the requested synchronized multi-M4A audio and transcript tool. The recommended approach is a **browser-based local web app with a Node.js/TypeScript backend that invokes native FFmpeg**. This gives the project the best balance of synchronized playback precision, local-only privacy, rich transcript navigation, non-destructive FFmpeg operations, maintainability, and time-to-value. Electron is a credible later packaging option if native desktop distribution becomes a hard requirement, but it adds embedded-browser security and release-maintenance overhead too early. A pure CLI plus generated HTML report is useful as a prototype or export mode, but it is not strong enough for interactive synchronization. A hosted/server-side transcoding web app is the best fit only if collaboration, centralized storage, or shared processing queues become more important than local-only handling of recorded calls.

## Options Identified

### Option 1: Browser-Based Local Web App With Web Audio API Plus Node/FFmpeg Backend

- **Description**: A local TypeScript backend runs on `localhost`, serves a browser UI, manages project/session state, validates JSONL transcript files, and invokes native FFmpeg/ffprobe through Node child processes. The frontend uses browser audio APIs for multi-track playback, transcript navigation, waveform/timeline views, selection state, manual offsets, and timestamp seeking. Audio processing outputs are generated by the backend into explicit derived files while preserving originals.
- **Strengths**:
  - Best balance for the requested interaction model: browser UI quality plus local native FFmpeg operations.
  - Web Audio provides an audio-context clock suitable for scheduling playback and timeline coordination [R1][R2][R3].
  - Native FFmpeg keeps merge, denoise, and loudness normalization close to the command-line filters the project already expects, including `afftdn`, `amerge`, `amix`, and `loudnorm` [R8].
  - Node's asynchronous `child_process.spawn()` model fits long-running FFmpeg tasks without blocking the backend event loop [R7].
  - Keeps audio and transcripts local by default, avoiding upload risk and server storage obligations.
  - Maintains a clean separation between UI concerns and privileged file/process operations.
  - Strong time-to-value because it avoids desktop packaging while still supporting rich transcript navigation.
- **Weaknesses**:
  - Requires the user to run a local service and open a browser, unless later wrapped.
  - Browser file access has permission and browser-support constraints. The File System Access API is useful for local file workflows but is not a universal browser capability, so the backend may need to own path-based file selection or the app may need to target Chromium explicitly [R4][R5].
  - High-precision multi-track playback may require decoding buffers or careful HTMLMediaElement/Web Audio coordination; long M4A files can create memory and seek-performance constraints.
  - Native FFmpeg availability must be detected explicitly, or a vetted packaged FFmpeg dependency must be chosen later.
- **Effort/Complexity**: Medium
- **Risk**: Medium
- **Best suited when**: The tool is primarily for one local operator reviewing call audio/transcripts, privacy matters, the UI must be rich, and FFmpeg operations should use native FFmpeg rather than browser WebAssembly.

### Option 2: Electron Desktop App

- **Description**: A packaged desktop TypeScript app using Electron, with a Chromium renderer for the UI and an Electron main process handling native file dialogs, file system access, and FFmpeg invocation. The renderer can still use Web Audio or HTML media elements for playback and timeline navigation.
- **Strengths**:
  - Gives the most seamless local-file user experience: native file/folder pickers, persistent paths, and desktop app behavior.
  - Can package a known FFmpeg binary, subject to licensing, platform, and dependency-vetting decisions.
  - Provides the same browser-class UI surface as Option 1 without asking the user to manually start a local server.
  - Good fit if this becomes a recurring desktop tool for non-technical users.
- **Weaknesses**:
  - Adds a larger security surface because the application combines browser rendering with privileged local APIs. Electron's own security guidance emphasizes context isolation, sandboxing, disabling unnecessary Node integration, validating navigation, and avoiding insecure content [R6].
  - Adds packaging, signing/notarization, auto-update, platform testing, and embedded Chromium maintenance.
  - Electron is specifically called out in the project dependency-vetting policy as a fast-moving embedded-engine dependency that must be carefully checked before adoption.
  - The architecture can be overbuilt for the current proposal phase and early local workflow.
- **Effort/Complexity**: High
- **Risk**: Medium-High
- **Best suited when**: The user needs a polished installable desktop app, non-technical distribution, robust native file access, and likely repeated use outside a developer environment.

### Option 3: Pure CLI Plus Generated HTML Report

- **Description**: A TypeScript CLI loads M4A and JSONL inputs, runs FFmpeg operations, normalizes transcript timestamps, and emits a static HTML report containing audio elements, transcript rows, links/bookmarks, and generated processing outputs. The HTML report can be opened locally in a browser.
- **Strengths**:
  - Fastest implementation path and easiest to test deterministically.
  - Very small runtime surface: TypeScript CLI plus FFmpeg and generated static HTML.
  - Strong fit for transcript presentation documents and repeatable batch exports.
  - Avoids long-lived server or Electron security posture.
  - Can become an export/report mode even if Option 1 is chosen as the main architecture.
- **Weaknesses**:
  - Weakest fit for the core interactive requirement: loading tracks dynamically, selecting subsets, manually correcting offsets, and controlling synchronized playback in real time.
  - Static HTML with multiple media elements can support approximate navigation, but it is less suitable for precise synchronized playback and drift correction than a live Web Audio-oriented workspace.
  - Transcript navigation is possible but would be report-centric rather than a full review workspace.
  - Re-running the CLI would be needed for many changes that a UI should handle interactively.
- **Effort/Complexity**: Low
- **Risk**: Low
- **Best suited when**: The priority is a quick, reproducible transcript/audio review artifact, not a fully interactive synchronization tool.

### Option 4: Web App Backed by Server-Side Transcoding

- **Description**: A conventional web application where users upload M4A and JSONL files to a backend service. The backend stores files, runs FFmpeg jobs, generates derived assets, and serves a browser UI for synchronized playback and transcript navigation. The server can be local, private LAN, or hosted cloud.
- **Strengths**:
  - Centralizes FFmpeg installation, job queues, storage, and generated outputs.
  - Best option for multi-user review, collaboration, shared archives, and long-running batch processing.
  - Easier to scale heavy FFmpeg operations with worker processes, containers, queues, and resource limits.
  - Browser UI can still provide the transcript navigation and timeline experience.
- **Weaknesses**:
  - Poor default fit for private recorded calls unless the server is strictly local or controlled. Upload workflows introduce file-upload security duties, storage lifecycle requirements, access control, and potential data-retention risks.
  - OWASP file-upload guidance calls for extension/type validation, safe filenames, storage controls, size limits, scanning or content validation where appropriate, and protection against parser and storage abuse [R10].
  - Adds infrastructure, authentication, authorization, logging/redaction, backup, cleanup, and operational monitoring concerns.
  - Time-to-value is lower for a single-user local tool.
- **Effort/Complexity**: High
- **Risk**: High
- **Best suited when**: Multiple reviewers need shared access, server-managed processing, persistent libraries, or centralized operations matter more than local-only privacy and low setup overhead.

## Recommendation

Recommend **Option 1: browser-based local web app with Web Audio API plus Node/FFmpeg backend** as the proposed architecture.

The deciding factors are:

- It matches the core workflow: local files, selected-track synchronized playback, transcript-driven seeking, and FFmpeg-derived outputs.
- It provides the best practical synchronization surface without committing to Electron's packaging and security overhead on day one.
- It keeps sensitive call recordings and transcripts local by default, avoiding the server-upload risk profile.
- It lets the frontend specialize in timeline/transcript UX while the backend handles privileged file and process operations.
- It can incorporate the CLI/report concept as an export mode without making that the main user experience.
- It leaves a migration path to Electron if later requirements demand a packaged desktop app.

The recommendation would change if:

- **Installable desktop app is mandatory for non-technical users**: choose Electron, but only after dedicated Electron security and packaging research.
- **The first deliverable is only a transcript/audio report, not an interactive review workspace**: choose CLI plus generated HTML as an initial MVP.
- **Shared multi-user review and centralized processing are mandatory**: choose server-side transcoding, but treat privacy, authentication, retention, and upload security as first-class requirements.
- **No local Node service is acceptable**: re-evaluate Electron or a pure CLI/report shape.

## Technical Research Guidance

**Research needed**: Yes

### Topic 1: Web Audio Multi-Track Synchronization for Long M4A Files

- **Why**: The recommended architecture depends on browser playback being reliable enough for selected multi-track synchronized review.
- **Focus**: AudioContext clock usage, AudioBufferSourceNode scheduling, pause/resume/seek semantics, HTMLMediaElement plus Web Audio tradeoffs, M4A decoding behavior, memory limits for long calls, drift detection, manual offset correction, and expected synchronization tolerance.
- **Depth**: Deep dive
- **Relevance**: Determines whether the frontend should decode complete tracks, stream through media elements, pre-render aligned derivatives, or combine approaches.

### Topic 2: FFmpeg Filter Chains for Merge, Denoise, and Level Calibration

- **Why**: The tool's audio operations are a core requirement and must be non-destructive, repeatable, and explainable.
- **Focus**: `ffprobe` metadata extraction, `amix` versus `amerge` versus `pan`, mono/stereo/multichannel export choices, `afftdn` denoising presets, `loudnorm` one-pass/two-pass EBU R128 normalization, progress reporting, temporary files, overwrite prevention, and output naming.
- **Depth**: Deep dive
- **Relevance**: Defines the backend job model and the user-facing processing/export options.

### Topic 3: Safe Local File Access and Localhost Backend Boundaries

- **Why**: Option 1 must bridge browser UI permissions and Node file/process privileges without creating unsafe path or command execution behavior.
- **Focus**: Browser file picker capabilities, File System Access API support limits, backend-owned file dialogs or import folders, path validation, workspace/session manifests, localhost CSRF/CORS boundaries, command argument escaping, use of `spawn()` without shell interpolation, and explicit missing-config errors.
- **Depth**: Intermediate to deep
- **Relevance**: Determines the boundary between frontend file handles and backend filesystem operations.

### Topic 4: Transcript JSONL Schema Normalization and Timestamp Mapping

- **Why**: Existing JSONL uses wall-clock ISO timestamps and durations, while audio navigation needs recording-relative offsets.
- **Focus**: Schema validation, line-level error reporting, required versus optional fields, wall-clock-to-audio-relative alignment, handling mixed languages, speaker/source grouping, search indexing, timestamp formatting, transcript row grouping, and linking sidecar Q&A records to the playback timeline.
- **Depth**: Intermediate
- **Relevance**: Drives the transcript model consumed by the playback workspace and generated reports.

### Topic 5: Waveform and Timeline Rendering Options

- **Why**: The proposal needs usable navigation for multi-track recordings and transcript-linked timestamps.
- **Focus**: Waveform rendering libraries, peak generation strategy, large-file performance, regions/markers, zoom/pan, selected-track overlays, offset editing UI, accessibility, and whether peaks are generated client-side or via FFmpeg/backend.
- **Depth**: Intermediate
- **Relevance**: Informs UI design and performance planning.

### Topic 6: Optional Electron Packaging Path

- **Why**: Electron is not recommended for the first architecture, but it remains a likely future packaging path if installability becomes important.
- **Focus**: Secure preload design, context isolation, sandboxing, disabled Node integration in renderers, IPC validation, native file dialogs, packaged FFmpeg, auto-update strategy, macOS signing/notarization, and dependency-vetting obligations.
- **Depth**: Overview to intermediate
- **Relevance**: Keeps the architecture adaptable without committing to Electron too early.

## Implementation Considerations

- Treat Option 1 as a **local-first application**, not a hosted product. The default assumption should be that audio and transcripts never leave the machine.
- Design the backend as the single privileged boundary for FFmpeg, ffprobe, output writing, and path access.
- Do not use shell command strings for FFmpeg operations. Build argument arrays and execute with a child process API that avoids shell interpolation.
- Define a session manifest that records loaded tracks, transcript files, detected durations, offsets, derived outputs, and user corrections without modifying originals.
- Make synchronization tolerance explicit before planning implementation. The likely starting target is "perceptually synchronized for call review", but the exact tolerance should be confirmed.
- Treat transcript timestamp alignment as a separate model from transcript display. The display can show clean wall-clock or relative timestamps, while navigation should use normalized recording-relative offsets.
- Add generated HTML/Markdown transcript reports as an export feature rather than the primary architecture.
- Keep Electron out of the initial plan unless installable desktop distribution becomes a firm requirement.
- If server-side processing is later requested, require a separate privacy/security design covering upload validation, storage location, retention, authentication, authorization, and deletion.
- Before any dependency is added in later phases, apply the project's dependency-vetting process and document the result in `Issues - Pending Items.md`.

## Open Questions

1. What synchronization tolerance is acceptable: under 20 ms, 20-50 ms, 50-100 ms, or simply perceptually aligned for meeting review?
2. Should automatic offset detection be required, or is manual offset correction enough for the first version?
3. Should transcript timestamps be treated as wall-clock, audio-relative, or mixed depending on source?
4. Should merging default to a mixed mono/stereo output, a left/right source-separated file, or a multichannel file?
5. Which denoising level is expected: simple FFmpeg denoise presets, configurable FFmpeg filters, or AI/RNNoise-style denoising if available?
6. Should the first proposal include transcript editing, or should transcript text remain read-only with navigation/search only?
7. Is a local Node service acceptable, or must the finished tool launch as a single desktop app?
8. Should output files remain beside the source files, inside a project workspace folder, or in a user-selected export folder?
9. Should generated transcript presentation be HTML, Markdown, PDF, or an in-app/export pair?
10. Are there privacy requirements that prohibit any hosted/server-side architecture entirely?
